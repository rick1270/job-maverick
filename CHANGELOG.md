# Job Maverick Changelog

## Session 2026-07-21 (v1.0 — initial fork)

### Changes
- **v1.0 — Forked from `life_job` v2.20**, genericized for sharing with non-technical friends (Google Sheets + Claude API access, no CLI/VS Code):
  - Bootstrapped a fresh Apps Script + Sheet project via `clasp create-script --type sheets` (script ID `1dWHxPy_B7NaoCIg8aUuhYz1pJvE-Rs9t7uT1lcSsvnIZS9vLOa5_dnsS`), separate from `life_job`'s own project — friends get their copy via **File > Make a Copy** on the Sheet, never touching clasp themselves.
  - Copied `Code.js`/`appsscript.json`/`sidebar.html`/`training_sidebar.html` as-is (no Rick-specific hardcoding found in Code.js beyond one file-naming default — the "Config is source of truth" architecture already fork-friendly).
  - Read and categorized the live `life_job` Config tab (107 rows): ~10 pure-personal fields (name/contact/resume URL), ~15 personal career-narrative rows, ~75 generic scoring/formatting methodology rows needing only "Rick"→neutral-phrasing wording changes (not `{{TOKEN}}` substitution, since most Config rows are dumped verbatim into the system prompt with no template engine — only a few specific rows already get real token substitution and keep it unchanged).
  - New `seedGenericConfigDefaults()`: one consolidated idempotent seed function covering all ~65 genericized methodology rows, including what were three separate `life_job` seed functions (`addTailoringConfigRows()`, `addFileNamingConfigRow()`, `addInterviewPrepConfigRow()`) — merged into one so the fork has a single source of truth instead of three overlapping ones. This mattered in practice: the initial fork copied `addTailoringConfigRows()` verbatim (Rick's literal wording intact) *alongside* the new genericized rows for the same Config keys, which would have silently seeded Rick's text first if it ran — caught and fixed by re-grepping for "Rick" after the first pass and deleting the redundant originals. Omitted rows that were Rick-specific role-strategy refinements rather than generalizable methodology (role-specific scoring sub-rules, narrowly-scoped gap rules naming specific tools/regions, the granular "Rick background: ..." row cluster — consolidated into the single `User Background` field the Setup Wizard collects). File Naming Pattern's default changed from `"RClayton_{{COMPANY}}_{{TYPE}}"` to `"{{COMPANY}}_{{TYPE}}"` (no hardcoded initials).
  - New **Setup Wizard** (`setup_wizard.html`, `showSetupWizard()`, `saveSetupWizardAnswers()`): a sidebar form (same `HtmlService.createTemplateFromFile` → `showSidebar` pattern as the existing Claude Chat/Training Notes sidebars, not a new modal-dialog paradigm) collecting personal info, resume/docs links, target role, background, and preferences, writing them into Config via `setConfigValue()`; also sets `CLAUDE_API_KEY` and auto-generates a `WEBHOOK_TOKEN` (no need for a non-technical user to invent one).
  - New **dynamic bookmarklet generation**: `bookmarklet_template.html` holds the bookmarklet JS pre-minified (single line, comments stripped by hand since Apps Script can't run an npm minifier) with `__BASE_URL__`/`__WEBHOOK_TOKEN__` placeholders — kept as a `.html` file specifically so Apps Script treats it as a static asset instead of trying to execute it as server code (a plain `.js` file would be parsed as GAS code and fail, since it uses `document`/`window`/browser APIs). `getBookmarkletSnippet()` substitutes the deployment's own URL (`ScriptApp.getService().getUrl()`) and webhook token at runtime using `JSON.stringify`-safe substitution (not raw concatenation, to avoid a stray quote breaking the string literal); `showBookmarkletDialog()` (`Job Assistant → Get Bookmarklet`) shows both a draggable link and a copy-paste fallback.
  - Swept remaining literal "Rick" references left over from the fork: two that actually reach Claude's prompts (a tool-schema description, a resume-fallback string) and four in code comments — all genericized to neutral third-person phrasing.
  - New docs: this file, README.md (dev-facing, mirrors `life_job`'s structure), CLAUDE.md (dev-facing, includes a documented `git remote` + `cherry-pick` workflow for porting mechanical fixes between the two separate repos), and SETUP_GUIDE.md (friend-facing, zero CLI/git references — Google Sheets UI + the in-browser Apps Script editor only).
